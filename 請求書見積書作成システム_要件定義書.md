## 1. プロジェクト概要

### 1.1 システム名称

請求書・見積書作成システム（MCP連携対応版）

### 1.2 プロジェクト背景・目的

IS Holdingsグループにおける見積書・請求書の作成から入金確認までの一連の帳票業務を、AIアシスタント（Claude）およびSlack等のコミュニケーションツールからMCPプロトコルを通じて半自動化することを目的とする。

従来の帳票作成業務では、手動でのデータ入力、PDF出力、入金確認といった定型作業に多くの時間が費やされている。本システムでは、MCP（Model Context Protocol）サーバーを構築し、Claude やSlack からの自然言語による指示で帳票の作成・管理を可能にすることで、業務効率の大幅な改善を実現する。

### 1.3 システム利用形態

| 項目 | 内容 |
|------|------|
| システム種別 | 社内業務システム |
| 利用企業 | IS Holdings（1社専用） |
| 利用者 | IS Holdingsグループ社員 |
| アクセス方法 | 1. Webアプリケーション（直接操作）<br>2. Claude（MCPサーバー経由）<br>3. Slack（MCPサーバー経由） |

### 1.4 用語定義

| 用語 | 定義 |
|------|------|
| MCP | Model Context Protocol。AIアシスタントが外部システムと連携するためのプロトコル |
| MCPサーバー | 本システムが公開するMCPエンドポイント。Claude/Slackからの操作を受け付ける |
| 帳票 | 見積書・請求書の総称 |
| 明細行 | 帳票内の個別品目（品名、数量、単価、金額等） |
| 承認フロー | 帳票の作成・送付・入金確認等において人間が最終確認を行うプロセス |
| 適格請求書 | インボイス制度（適格請求書等保存方式）に基づく法定記載事項を満たした請求書 |
| 登録番号 | 適格請求書発行事業者登録番号（T+13桁の法人番号） |
| 電子帳簿保存法 | 電子データで作成・保存する帳簿書類に関する法律。PDF保存時の要件を規定 |
| 源泉徴収 | 報酬等の支払時に所得税を差し引いて納付する制度 |

## 3. 法令・制度対応

### 3.1 インボイス制度（適格請求書等保存方式）

2023年10月施行の適格請求書等保存方式に準拠する。本システムで作成する請求書・見積書は、以下の法定記載事項を満たさなければならない。

#### 3.1.1 法定記載事項

| No. | 記載事項 | 対応箇所 |
|-----|----------|----------|
| 1 | 適格請求書発行事業者の氏名又は名称 | 自社情報（F-006） |
| 2 | 適格請求書発行事業者の登録番号（T+13桁） | 自社情報（F-006）に追加 |
| 3 | 取引年月日 | 発行日 |
| 4 | 取引内容（軽減税率対象品目である旨） | 明細行の税区分に「※」等で表示 |
| 5 | 税率ごとに区分して合計した対価の額（税抜又は税込） | 合計欄に税率別内訳を表示 |
| 6 | 税率ごとに区分した消費税額等 | 合計欄に税率別消費税額を表示 |
| 7 | 書類の交付を受ける事業者の氏名又は名称 | 宛名（取引先名） |

#### 3.1.2 PDF出力への反映

- PDFテンプレートに登録番号の表示欄を設ける
- 軽減税率（8%）対象の明細行には「※」マークを付与し、欄外に「※は軽減税率対象」と注記
- 合計欄に税率ごとの内訳（10%対象小計・消費税、8%対象小計・消費税）を表示
- 見積書は適格請求書の法定要件の対象外だが、請求書変換時の整合性のため同等のレイアウトを採用

### 3.2 電子帳簿保存法対応

本システムで電子的に作成・保存する帳票は、電子帳簿保存法（電子取引データ保存）の要件に準拠する。

| 要件 | 対応方針 |
|------|----------|
| 真実性の確保（改ざん防止） | S3バケットのオブジェクトロック（WORM）を有効化。確定済みPDFは上書き・削除不可とする。帳票取消時も原本PDFは保持し、取消版PDFは別オブジェクトとして派生保存する（F-017 4.16.1節参照） |
| タイムスタンプ | PDF生成時にタイムスタンプを付与（認定タイムスタンプサービスの利用を検討） |
| 検索要件 | 取引年月日・取引先名・取引金額での検索機能を帳票一覧画面（F-004）で提供 |
| 見読性の確保 | Web画面でのプレビュー表示およびPDFダウンロード機能で担保 |
| 保存期間 | 法定保存期間（7年、繰越欠損金がある場合は10年）に準拠。詳細は8.6節を参照 |

### 3.3 源泉徴収税の取り扱い

| 項目 | 内容 |
|------|------|
| 対応要否 | Phase 1では対応しない。Phase 4以降で対応を検討する |
| 対象ケース | フリーランス・個人事業主への報酬支払い（デザイン費、コンサルティング費等） |
| 計算ロジック（将来対応） | 報酬額100万円以下: 10.21%、100万円超: 20.42%（超過分） |
| 帳票への影響（将来対応） | 明細行または合計欄に源泉徴収税額の表示欄を追加。差引支払額を計算・表示 |

> **注記:** 源泉徴収対応は取引先の属性（法人/個人）に依存するため、取引先マスタに事業者区分の属性を追加する必要がある（7.3節参照）。

### 4.2 見積書作成機能（F-001）

#### 4.2.1 入力項目

| カテゴリ | 項目 | 型 | 必須 |

| ヘッダー | 有効期限 | Date | ○ |
| ヘッダー | 件名 | String | ○ |
| 取引先 | 取引先（選択） | Select | ○ |
| 取引先 | 宛名（自動入力） | String | ○ |
| 明細行 | 品名 | String | ○ |
| 明細行 | 数量 | Number | ○ |
| 明細行 | 単位 | String | － |
| 明細行 | 単価 | Number | ○ |
| 明細行 | 税区分 | Select | ○ |
| 明細行 | 金額（自動計算） | Number | ○ |
| 合計 | 小計 | Number（自動） | ○ |
| 合計 | 消費税 | Number（自動） | ○ |
| 合計 | 合計金額 | Number（自動） | ○ |
| 備考 | 備考欄 | Text | － |

#### 4.2.2 機能仕様

- 明細行は動的に追加・削除・並び替えが可能
- 税区分は「10%」「8%（軽減税率）」「非課税」から選択
- 金額は「数量 × 単価」で自動計算
- 消費税は税区分ごとに集計し、端数処理は「切り捨て」を既定とする
- 端数処理の適用単位: 税率区分ごとの合計金額に対して1回適用する（明細行単位ではない）
- 合計欄には税率ごとの内訳（10%対象小計・消費税、8%対象小計・消費税）を表示する（インボイス制度対応、3.1節参照）
- 見積書番号は `EST-YYYYMMDD-NNN` 形式で自動採番
- 採番はDynamoDBの条件付き書き込み（`ConditionExpression`）により排他制御し、同時採番時の重複を防止する
- 下書き保存・確定の2段階ステータスを持つ
- 確定後のPDF出力が可能

### 4.3 請求書作成機能（F-002）

見積書と同等の入力項目に加え、以下の項目を追加する。

| カテゴリ | 項目 | 型 | 必須 |
|----------|------|----|------|
| 支払 | 支払期限 | Date | ○ |
| 支払 | 振込先情報（自動） | String | ○ |

- 請求書番号は `INV-YYYYMMDD-NNN` 形式で自動採番
- 請求書の業務進行は帳票ステータス（`draft` → `pending_approval` → `confirmed` → `sent` → `paid` / `overdue` / `cancelled`）で一元管理する。独立した「入金ステータス」属性は持たない（4.4節参照）
- `sent`（送付済）が旧来の「請求済」に、`paid`（入金済）が「入金済」に、`overdue`（遅延）が「遅延」にそれぞれ対応する。`draft` / `pending_approval` / `confirmed` が「未請求」に相当する
- 見積書から請求書への変換時、明細行データを自動コピー
- 端数処理は見積書と同様、税率区分ごとの合計金額に対して「切り捨て」を適用
- 適格請求書の法定記載事項（登録番号、税率別内訳等）をPDFに出力（3.1節参照）

### 4.4 帳票ステータス管理

#### 4.4.1 ステータス遷移

| 帳票種別 | ステータス遷移 |
|----------|----------------|
| 見積書 | 下書き（`draft`）→ 承認待ち（`pending_approval`）→ 確定（`confirmed`）→ 送付済（`sent`）→ 受注（`won`）/ 失注（`lost`）/ 取消（`cancelled`） |
| 請求書 | 下書き（`draft`）→ 承認待ち（`pending_approval`）→ 確定（`confirmed`）→ 送付済（`sent`）→ 入金済（`paid`）/ 遅延（`overdue`）/ 取消（`cancelled`） |

#### 4.4.2 ステータス値定義

| ステータス値 | 日本語名 | 対象帳票 | 説明 |
|-------------|---------|---------|------|
| `draft` | 下書き | 見積書・請求書 | 作成中。編集・削除可能 |
| `pending_approval` | 承認待ち | 見積書・請求書 | 作成者が承認依頼済み。承認者（作成者以外の経理担当者/管理者）による承認または差戻しを待機中。内容の編集不可（差戻し後に `draft` に戻り編集可能） |
| `confirmed` | 確定 | 見積書・請求書 | 承認済み。内容の編集不可（改訂が必要な場合はF-018を使用） |
| `sent` | 送付済 | 見積書・請求書 | 取引先に送付済み |
| `won` | 受注 | 見積書のみ | 受注確定 |
| `lost` | 失注 | 見積書のみ | 失注 |
| `paid` | 入金済 | 請求書のみ | 入金確認済み |
| `overdue` | 遅延 | 請求書のみ | 支払期限超過 |
| `cancelled` | 取消 | 見積書・請求書 | 確定後に取り消された帳票。取消理由を記録する |

> **注記:** `won`/`lost` は見積書専用、`paid`/`overdue` は請求書専用のステータスである。データベース上は全ステータス値を `status` 属性に格納するが、UIおよびバリデーションで帳票種別ごとに利用可能なステータスを制限する。

### 4.5 見積書→請求書変換機能（F-003）

#### 4.5.1 変換仕様

| 項目 | 仕様 |
|------|------|
| 変換元ステータス | `confirmed` または `sent` または `won` の見積書のみ変換可能 |

| 件名 | 件名 | そのままコピー |

### 4.10 入金確認連携機能（F-011）

#### 4.10.1 照合仕様

| 項目 | 仕様 |
|------|------|
| 照合対象 | ステータスが `sent` または `overdue` の請求書 |
| 照合キー | 請求書の合計金額 + 取引先名（振込名義との部分一致） |
| 照合方式 | マネーフォワード等から入金データを取得し、自動照合 |
| 照合精度 | 金額完全一致 + 振込名義の部分一致（カナ変換後に比較） |

#### 4.10.2 照合結果

| 結果 | 条件 | アクション |
|------|------|-----------|
| 完全一致 | 金額一致 & 名義一致 | ユーザーに確認提示 → 承認後 `paid` に更新 |
| 金額不一致 | 名義一致だが金額が異なる | 差額を提示し、部分入金か誤入金かをユーザーが判断 |
| 部分入金 | 請求金額の一部のみ入金 | 入金額を記録し、残額を管理。ステータスは `sent` のまま |
| 未入金 | 照合対象の入金データなし | ステータス変更なし。支払期限超過時は `overdue` に自動更新 |

#### 4.10.3 実行方式

- **手動実行:** Webアプリまたは MCP ツール `check_payment` から随時実行可能
- **定期実行:** 平日10:00にバッチ処理で全 `sent` / `overdue` 請求書を一括照合（EventBridge + Lambda）
- **結果通知:** 照合結果をSlack通知（5.2節）で経理担当者に送付

### 4.11 承認フロー機能（F-012）

#### 4.11.1 承認モデル

| 項目 | 仕様 |
|------|------|
| 承認方式 | 単段承認（承認者1名による承認で完了） |
| 承認経路 | 帳票作成者が承認を依頼 → **作成者以外の**承認権限を持つユーザー（経理担当者 or 管理者）が承認 |
| 自己承認 | **不可**。作成者と承認者は必ず異なるユーザーでなければならない（四眼原則）。システムで強制する |
| 代理承認 | Phase 1では非対応。将来的に代理承認者の設定機能を検討 |
| 承認期限 | 承認依頼から72時間以内。期限超過時にリマインド通知 |

> **設計原則:** 本システムでは四眼原則（作成者 ≠ 承認者）を採用する。帳票の確定・送付・入金ステータス更新・取消のいずれも、操作を起案した本人が自ら承認することはできない。これにより、不正防止と監査証跡の信頼性を確保する。

#### 4.11.2 承認操作

| 操作 | 説明 |
|------|------|
| 承認（approve） | 帳票を次のステータスに遷移させる。承認者は作成者以外であること |
| 差戻し（reject） | 帳票を `draft` に戻す。差戻し理由（必須）をコメントとして記録 |

#### 4.11.3 承認履歴

- 全承認操作の履歴を記録: 承認者、操作日時、操作（approve/reject）、コメント
- 帳票詳細画面から承認履歴を閲覧可能
- 監査ログにも記録（7.4節参照）

### 4.15 帳票複製機能（F-016）

- 既存の帳票（見積書・請求書）をベースに新規の下書き帳票を作成する
- 複製時、帳票番号は新規採番。発行日は当日に設定。ステータスは `draft`
- 明細行・取引先・件名・備考はすべてコピーされる
- 有効期限（見積書）/ 支払期限（請求書）はユーザーが再設定する

### 4.16 帳票取消・無効化機能（F-017）

- `confirmed` / `sent` ステータスの帳票を取消（`cancelled`）に遷移させる
- 取消理由（テキスト入力、必須）を記録する
- 取消には経理担当者または管理者の承認が必要（四眼原則適用）

#### 4.16.1 取消時のPDF管理（電子帳簿保存法との整合）

| PDF種別 | S3パス | 扱い |
|---------|--------|------|
| 原本PDF | `documents/{documentId}/original.pdf` | **変更・削除不可**（S3 Object Lock適用済み）。電子帳簿保存法の真実性確保のため、取消後も永続保存 |
| 取消版PDF | `documents/{documentId}/cancelled.pdf` | 原本PDFを複製し「取消」の透かし（ウォーターマーク）を追加した**別オブジェクト**として新規保存 |

- 原本PDFは一切上書き・削除しない。取消版PDFは原本からの派生物として別キーで保存する
- 帳票詳細画面では取消版PDFを既定表示し、原本PDFへのリンクも提供する（「原本を表示」ボタン）
- データベース上は `pdfUrl`（原本）と `cancelledPdfUrl`（取消版）の2属性で管理する
- 取消済み帳票は一覧画面でグレーアウト表示。検索フィルタで除外可能

### 4.17 帳票改訂機能（F-018）

- `confirmed` / `sent` ステータスの見積書に対し、改訂版を作成する
- 改訂版は元の見積書の内容をコピーし、`draft` として新規作成
- 見積書番号は `EST-YYYYMMDD-NNN-R{revision}` 形式（例: `EST-20260327-001-R2`）
- 元の見積書には `revisedByDocumentId` を記録し、改訂版との関連を保持
- 改訂版には `originalDocumentId` と `revisionNumber` を記録

### 5.3 MCP連携フロー

#### 5.3.1 見積書作成フロー（Claude/Slack経由）

| Step | アクター | アクション | 補足 |
|------|----------|------------|------|
| 1 | ユーザー | Claude/Slackで見積書作成を依頼 | 例:「A社に開発費用100万円の見積書作って」 |
| 2 | Claude/Bot | MCPツール `create_estimate` を呼び出し | 自然言語からパラメータを抽出 |
| 3 | MCPサーバー | バリデーション後、DynamoDBに下書き保存 | 取引先名から取引先IDを自動解決 |
| 4 | MCPサーバー | 作成結果をレスポンス | 帳票ID、概要、下書きステータスを返却 |
| 5 | Claude/Bot | ユーザーに内容確認を提示 | 金額、明細、取引先等の確認 |
| 6 | ユーザー | 内容を確認し、承認依頼を指示 | 「OK、承認に回して」等の応答 |
| 7 | Claude/Bot | MCPツール `update_status` で承認依頼を送信 | ステータスを `draft` → `pending_approval` に更新。承認権限を持つユーザー（経理担当者/管理者）にSlack通知 |
| 8 | 承認者 | 承認一覧画面またはSlackで承認操作 | **作成者以外**の経理担当者/管理者が `approve_document` を実行（四眼原則） |
| 9 | MCPサーバー | ステータスを `confirmed` に更新、PDF生成・保存 | S3にPDFを保存 |
| 10 | Claude/Bot | 作成者に確定通知とPDFを返却 | ダウンロードリンクまたはファイル添付 |

#### 5.3.2 入金確認フロー

| Step | アクター | アクション | 補足 |
|------|----------|------------|------|
| 1 | システム/ユーザー | 入金確認を実行 | 定期バッチまたは手動トリガー |
| 2 | MCPサーバー | マネーフォワード等のMCPサーバーに照会 | 請求書の金額・取引先で照合 |
| 3 | MCPサーバー | 照合結果を返却 | 一致/不一致/部分入金を判定 |
| 4 | Claude/Bot | ユーザーに確認結果を提示 | 「A社の請求書INV-xxx: 入金確認済み」 |
| 5 | ユーザー | 承認（入金確定） | ステータスを「入金済」に更新承認 |

### 5.4 承認フローの設計方針

本システムでは「半自動」を基本方針とし、すべての重要アクションにおいて人間の承認を必須とする。

| 操作 | 承認要否 | 承認者 |
|------|----------|--------|
| 帳票の新規作成（下書き） | 不要（自動保存） | － |
| 帳票の確定 | 必要 | 経理担当者 / 管理者（作成者以外。四眼原則） |
| 帳票の送付 | 必要 | 経理担当者 / 管理者 |
| 入金ステータスの更新 | 必要 | 経理担当者 / 管理者 |
| 帳票の削除 | 必要 | 管理者 |
| 帳票の取消 | 必要 | 経理担当者 / 管理者 |

### 5.5 MCPエラーハンドリング

#### 5.5.1 エラーレスポンス形式

MCPツールのエラーレスポンスは以下の形式で統一する。

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "取引先 'ABC商事' が見つかりません。",
    "details": {
      "field": "client_name",
      "suggestion": "登録済みの取引先: ABC商事株式会社, ABC商事（大阪）"
    }
  }
}
```

#### 5.5.2 エラーコード一覧

| エラーコード | 説明 | 対応 |
|-------------|------|------|
| `VALIDATION_ERROR` | 入力パラメータのバリデーションエラー | エラー内容と修正候補を返却 |
| `NOT_FOUND` | 指定されたリソースが存在しない | 対象リソースのIDと種別を返却 |
| `PERMISSION_DENIED` | 操作権限がない | 必要な権限と現在の権限を返却 |
| `INVALID_STATUS` | 現在のステータスでは許可されない操作 | 現在のステータスと許可される操作を返却 |
| `DUPLICATE_REQUEST` | 重複リクエスト（冪等キーが既に処理済み） | 既存の処理結果を返却 |
| `EXTERNAL_SERVICE_ERROR` | 外部サービス（マネーフォワード等）のエラー | リトライ可否と推奨待機時間を返却 |
| `INTERNAL_ERROR` | 内部エラー | エラーIDを返却（詳細はログで確認） |

### 5.6 冪等性・リトライ

| 項目 | 仕様 |
|------|------|
| 冪等キー | 作成系ツール（`create_estimate`, `create_invoice`, `create_client`）はリクエストヘッダーに `Idempotency-Key` を含めることができる |
| 冪等キーの有効期間 | 24時間。同一キーでの再リクエストは同一結果を返却する |
| 保存先 | DynamoDB に冪等キーと処理結果を保存 |
| リトライポリシー | MCPクライアント側で指数バックオフ（初回1秒、最大30秒、最大3回）を推奨 |
| 読み取り系ツール | `get_document`, `list_documents`, `list_clients`, `get_dashboard` は冪等キー不要（自然に冪等） |

### 5.7 MCP認証詳細

| 項目 | 仕様 |
|------|------|
| 認証方式 | OAuth 2.0（Authorization Code Flow with PKCE）を採用 |
| トークン発行 | AWS Cognito をOAuth 2.0プロバイダーとして使用 |
| アクセストークン有効期間 | 1時間 |
| リフレッシュトークン有効期間 | 30日 |
| スコープ | `documents:read`, `documents:write`, `clients:read`, `clients:write`, `admin` |
| Claude連携 | Claude MCP設定でOAuth認証情報を設定。ユーザーのCognito認証情報に基づきアクセス制御 |

## 13. 付録

### 13.1 見積書 PDF レイアウト（イメージ）

標準的な日本式見積書レイアウトを採用する。以下の構成要素を含む。

- タイトル:「御見積書」/「御請求書」（中央上部）
- 発行日、帳票番号（右上）
- 宛名: ○○株式会社 御中（左上）
- 自社情報: 会社名、住所、電話番号、**登録番号（T+13桁）**、印影（右側）
- 件名、有効期限（見積書）/ 支払期限（請求書）
- 明細テーブル: No. / 品名 / 数量 / 単位 / 単価 / 金額 / 税区分
- **税率別内訳: 10%対象（小計・消費税）、8%対象（小計・消費税）**
- 小計、消費税合計、合計金額
- 備考欄
- **軽減税率注記（該当がある場合）:「※は軽減税率（8%）対象」**
- 振込先情報（請求書の場合）

> 上記レイアウトは適格請求書（インボイス）の法定記載事項（3.1節）を満たす。
